# MingSui 产品化路线图

MingSui 的目标不是只做一个能跑的代理工具，而是做成一个桌面端、CLI 和 AI 工作流一致的代理产品。桌面端承担可视化控制、订阅维护、诊断和日常使用；CLI 承担脚本化、终端环境注入、AI Agent 调用和服务器侧运维。两端必须共享同一份配置、同一套概念和同一条排障路径。

参考方向是 Clash Party 一类成熟代理客户端的产品感：信息密度高、状态明确、操作入口稳定、视觉克制，用户打开后能马上知道当前是否可用、正在使用哪个节点、下一步该做什么。MingSui 的差异是把 AI/CLI 场景放到一等位置，不能只做传统机场订阅管理。

## 设计原则

- 桌面端必须优先呈现运行状态、当前节点、本地入口、系统代理和问题提示。
- 主要页面采用工具型布局，不做营销式首页，不用大面积装饰背景。
- 每个页面都要有清晰的空状态、错误状态、处理中状态和成功反馈。
- 所有危险或影响网络的操作要有明确提示，例如删除节点、跳过 TLS 校验、启用系统代理。
- 桌面端和 CLI 使用同一套术语：节点、订阅、Relay、SOCKS5、HTTP、系统代理、AI 环境变量。
- 设置项按用户心智分区，而不是按配置 JSON 字段堆叠。
- 视觉上保持浅色和暗色都可用，组件半径、按钮尺寸、表格密度保持一致。

## 阶段一：设置中心

设置页当前最像 demo，因为它只是把配置字段铺在页面上。第一阶段要把它重构成设置中心：

- 顶部状态卡：展示 SOCKS5、HTTP、Relay、保存状态。
- 左侧配置目录：本地入口、Relay 默认连接、安全认证、TLS 传输，显示每组的启用状态和关键值。
- 右侧配置表单：按分区组织输入项，减少用户理解成本。
- 表单校验：本地监听地址、HTTP 地址、超时秒数、启用认证后的用户名密码必须可解释。
- 安全提示：默认 token、跳过 TLS 校验、启用 TLS 但缺少 ServerName 或 CA 文件都要显示提示。
- 保存边界：没有致命错误时才能保存；保存完成后显示反馈并清除未保存状态。

完成标准：

- 前端构建通过。
- 原有保存 API 不变，继续保留 profiles、proxy_profiles、subscriptions。
- 轮询刷新不会覆盖用户正在编辑的未保存内容。
- 暗色模式下主要文本、输入框、状态提示可读。

## 阶段二：总览控制台

总览页要成为产品首页，而不是功能集合：

- 连接卡展示当前模式、当前节点、Relay 地址、系统代理状态、流量和连接数。
- readiness 提示要变成可执行任务区，直接引导导入订阅、选择节点、开启系统代理或复制 AI 命令。
- AI Agent 工具区要展示可复制的 `mingsui env`、`mingsui exec -connect -- <command>` 和代理变量。
- 快速导入要保留订阅、节点文本和默认选择逻辑，同时补齐导入前后状态。

完成标准：

- 用户不看 README 也能完成连接、导入、选优和复制终端命令。
- CLI 命令从桌面复制后可直接执行。
- 系统代理不可用时显示原因和替代方案。

## 阶段三：节点和订阅工作台

节点和订阅要从列表升级为维护工作台：

- 节点页按可连接、当前、国内、不支持分类，并保留协议统计。
- 节点操作包含选择、测速、删除、自动选优，并明确不可自动选择的原因。
- 订阅页展示来源、同步策略、上次同步结果、导入数量和错误信息。
- 订阅同步后要能直接进入节点页确认候选节点。

完成标准：

- 大量节点时仍可搜索、过滤、扫描。
- 订阅失败时能看到失败来源和下一步建议。
- 节点兼容性与 CLI 导出能力一致。

## 阶段四：诊断中心

日志页要升级成诊断中心，而不是纯文本输出：

- 日志支持级别过滤、关键词搜索、复制和清空视图。
- readiness、系统代理、配置路径、当前进程状态集中展示。
- 常见问题提供一键复制诊断命令，例如 `mingsui doctor -json`、`mingsui status`、`mingsui config proxy list -json`。
- 关键错误要能关联到配置项或页面入口。

完成标准：

- 运行失败时，用户能从桌面端找到可执行的排障步骤。
- CLI 诊断输出和桌面 readiness 信息保持一致。

## 阶段五：CLI 与桌面闭环

成品形态要求 CLI 和桌面互相补位：

- CLI 帮助文本中文优先，命令分组清晰。
- 面向 AI 和脚本的导入、同步、列表、选择和诊断命令都有稳定 JSON 输出。
- `mingsui env` 和 `mingsui exec` 是 AI/终端场景的主路径。
- `mingsui import -check`、`select-best`、`doctor` 和桌面节点/诊断能力一致。
- 发布包包含明确的安装、启动、升级、卸载路径。
- 桌面端启动文档区分开发启动、源码构建和用户安装包启动，避免把网页预览当成客户端。

完成标准：

- `go test ./...` 和前端构建通过。
- 冒烟测试覆盖 CLI、配置、订阅导入、代理环境变量和连接路径。
- 每个可发布切片都形成小 commit，便于回滚和审查。
